# 群防群治智慧警务平台 - 项目文档

## 一、项目简介

群防群治智慧警务平台是一个基于 Flask 框架开发的警民联动 Web 应用，旨在整合社会力量，构建"专业+机制+大数据"的新型警务运行模式。平台以"人人都是平安合伙人，处处都是防控第一线"为核心理念，通过悬赏令发布、线索举报、安全隐患上报、任务抢单、积分激励等功能模块，实现警务工作与社会力量的深度融合。

### 项目信息

| 属性 | 内容 |
|------|------|
| 项目名称 | 群防群治智慧警务平台 |
| 开发语言 | Python 3.8+ |
| Web框架 | Flask 3.0 |
| 数据库 | SQLite（开发）/ MySQL（生产，可选） |
| 前端技术 | HTML5 + CSS3 + JavaScript + Bootstrap 5.3 |
| 地图服务 | 高德地图 API v2.0 |
| 版本控制 | Git |
| 开源协议 | MIT |

## 二、功能说明

### 2.1 用户系统

平台支持多种用户角色，每种角色拥有不同的功能权限：

| 角色 | 说明 | 核心权限 |
|------|------|----------|
| 普通群众 (citizen) | 默认注册角色 | 查看悬赏令、提交线索、上报隐患 |
| 志愿者 (volunteer) | 通过注册升级 | 额外可抢单执行任务 |
| 安保人员 (security) | 专业安保力量 | 巡逻打卡、应急响应、位置上报 |
| 网格员 (grid_worker) | 社区管理者 | 社区管理、信息采集 |
| 云哨兵 (cloud_sentinel) | 流动信息员 | 流动人员管理、见疑即报 |
| 民警 (police) | 执法人员 | 线索处理、任务审核、应急调度 |
| 管理员 (admin) | 系统管理者 | 系统管理、用户管理、数据统计 |

用户注册后可通过"志愿者注册"页面升级角色，加入群防群治队伍。

### 2.2 悬赏令系统

- 管理员发布悬赏通告，支持刑事案件、治安案件、寻人启事、失物招领等多种类型
- 悬赏详情页展示嫌疑人信息、照片、案发地点（地图标注）
- 支持现金悬赏和积分悬赏两种方式
- 群众可在线提交线索，支持匿名举报
- 悬赏令状态流转：进行中 → 待审核 → 已结案/已取消

### 2.3 线索举报系统

- 支持关联特定悬赏令提交线索，也可独立举报
- 匿名举报模式：系统生成唯一查询码，举报人凭码追踪进度
- 拍照上传（支持多图），自动记录 GPS 位置信息
- AI 图像识别功能：上传照片自动识别隐患类型
- 线索状态可视化追踪：已提交 → 审核中 → 核查中 → 已核实 → 已办结 → 已奖励
- 实名举报用户可获得积分奖励，线索被采纳额外加分

### 2.4 安全隐患上报

- 随手拍功能：拍照上传隐患照片，自动定位
- 隐患分类：消防通道堵塞、井盖缺失、路灯损坏、路面塌陷、电线裸露、车辆违停等
- 紧急程度标记：低/一般/高/紧急
- 地图可视化展示：所有隐患在地图上标注，支持热力图
- 处理进度跟踪：待处理 → 处理中 → 已解决/已忽略
- 积分奖励：上报隐患 +10分，隐患被解决再 +20分

### 2.5 任务抢单系统

- 管理员发布任务：巡逻任务、宣传任务、应急任务、调查任务
- 设定任务地点（地图选点）、时间范围、所需人数、积分/现金奖励
- 志愿者在线抢单，先到先得
- 任务进度跟踪：可抢单 → 已接单 → 进行中 → 已完成 → 已审核
- 完成任务需上传证明照片，管理员审核后发放积分奖励

### 2.6 积分激励体系

- 积分获取途径：提交线索 +10、线索被采纳 +50、完成任务 +30、巡逻打卡 +5、应急响应 +100、上报隐患 +10、隐患被解决 +20
- 积分商城：使用积分兑换实物商品
- 积分排行榜：按累计积分排名，管理员不参与排行
- 积分变动记录：完整的积分明细日志

### 2.7 治安防控地图

- 隐患热力图：直观展示辖区安全隐患分布
- 在线安保人员定位：实时显示值班安保/网格员/云哨兵位置
- 应急调度可视化：在地图上发起应急事件，自动圈选附近可用力量
- 高德地图为基础，支持标记点击查看详情

### 2.8 应急调度系统

- 管理员创建应急事件，在地图上标注事件位置
- 设置调度半径，系统自动匹配范围内安保人员
- 安保人员收到应急通知后可一键响应
- 应急事件状态：调度中 → 响应中 → 已解决/已取消
- 参与应急响应获得高额积分奖励（+100分）

### 2.9 公告系统

- 发布通知公告、安全提示、新闻资讯
- 支持置顶、分类筛选
- 支持封面图片上传
- 浏览量统计

### 2.10 巡逻打卡

- 安保人员/网格员在巡逻过程中进行位置打卡
- 支持拍照上传、备注说明
- 自动记录打卡时间和位置
- 打卡后自动切换为"值班中"状态

## 三、设计思路

### 3.1 架构设计

采用经典的 MVC（Model-View-Controller）架构模式：

```
police-platform/
├── app/                    # 应用主目录
│   ├── __init__.py         # Flask 应用工厂
│   ├── models.py           # 数据模型层（Model）
│   ├── routes/             # 路由控制层（Controller）
│   │   ├── main.py         # 前台主路由
│   │   ├── auth.py         # 用户认证路由
│   │   ├── api.py          # RESTful API 路由
│   │   └── admin.py        # 管理后台路由
│   ├── templates/          # 视图模板层（View）
│   ├── static/             # 静态资源
│   └── utils/              # 工具函数
├── config/                 # 配置文件
├── docs/                   # 项目文档
├── requirements.txt        # Python 依赖清单
└── run.py                  # 应用启动入口
```

**应用工厂模式**：通过 `create_app()` 工厂函数创建 Flask 实例，支持通过环境变量切换开发/生产/测试三种配置，便于不同环境的部署。

**蓝图（Blueprint）模块化**：按功能将路由分为 main、auth、api、admin 四个蓝图，实现功能解耦和模块化管理。

**前后端不分离的混合架构**：前台页面使用 Flask 模板渲染（Jinja2）+ Bootstrap 5.3 响应式布局，管理后台部分功能使用 AJAX 异步交互提升体验。对于可能扩展的移动端需求，预留了完整的 RESTful API 接口。

### 3.2 数据库设计

数据库共设计 **13 张核心表**，覆盖业务全流程：

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| users | 用户表 | 用户名、密码哈希、手机号、角色类型、积分、位置信息、值班状态 |
| rewards | 悬赏令表 | 标题、案件类型、悬赏金额、嫌疑人信息、地理位置、状态 |
| clues | 线索举报表 | 线索编号、关联悬赏令、举报内容、匿名标记、位置、状态流转 |
| hazards | 安全隐患表 | 隐患编号、类型、描述、位置坐标、紧急程度、处理状态 |
| tasks | 任务表 | 任务编号、类型、时间范围、所需人数、积分/现金奖励 |
| task_assignments | 任务分配表 | 关联任务和用户、状态、完成证明、审核信息 |
| patrol_checkins | 巡逻打卡表 | 用户、打卡位置、照片、时间 |
| emergency_dispatches | 应急调度表 | 事件描述、类型、位置、响应人员、状态 |
| announcements | 公告表 | 标题、内容、分类、置顶标记、浏览量 |
| reward_items | 积分商品表 | 商品名称、所需积分、库存 |
| reward_claims | 兑换记录表 | 用户、商品、消耗积分、收货信息、物流 |
| point_logs | 积分日志表 | 用户、积分变动值、变动原因、时间 |
| system_configs | 系统配置表 | 键值对配置项 |

**设计原则**：
- 使用自增整数主键，业务编号（如线索编号 XS20240521...）用于对外展示
- 媒体文件（图片等）存储为 JSON 数组字符串，文件实体存储在服务器文件系统
- 用户密码使用 Werkzeug 提供的 `generate_password_hash` 进行哈希加密，不存储明文
- 位置信息使用经纬度（lat/lng）统一存储，与高德地图坐标系（GCJ-02）兼容

### 3.3 安全设计

- **密码安全**：所有用户密码使用 Werkzeug 哈希算法加密存储，无法逆向破解
- **SQL 注入防护**：使用 SQLAlchemy ORM 参数化查询，杜绝 SQL 注入
- **XSS 防护**：Jinja2 模板引擎默认进行 HTML 转义
- **文件上传安全**：三重验证机制 —— 客户端 MIME 检查 + 服务端扩展名白名单 + MIME 类型验证，仅允许 `png/jpg/jpeg/gif/webp` 图片格式
- **权限控制**：管理员路由使用 `@admin_required` 装饰器进行角色验证，普通用户无法访问
- **匿名举报保护**：匿名线索不关联用户 ID，仅凭系统生成的随机查询码追踪
- **敏感信息保护**：`.env` 文件已加入 `.gitignore`，不会提交到版本控制系统

### 3.4 积分系统设计

积分系统采用"当前积分 + 累计积分"双字段设计：
- `points`：当前可用积分（兑换后扣减）
- `total_points`：累计获得积分（用于排行榜排名）

每次积分变动都记录到 `point_logs` 表，保证积分账目可追溯。

## 四、使用指南

### 4.1 环境要求

- Python 3.8 及以上版本
- 操作系统：Windows / Linux / macOS 均可
- 内存：至少 512MB
- 硬盘：至少 1GB 可用空间

### 4.2 快速部署（一键启动）

本项目使用 **SQLite 数据库**，无需安装任何数据库软件，开箱即用。

```bash
# 1. 克隆项目
git clone https://github.com/Hyperspace138/police-platform.git
cd police-platform

# 2. 创建虚拟环境（推荐）
python3 -m venv venv
source venv/bin/activate    # Linux/Mac
# venv\Scripts\activate     # Windows

# 3. 安装依赖
pip install -r requirements.txt

# 4. 配置环境变量（可选，不配置也可运行）
cp .env.example .env

# 5. 启动应用
python run.py
```

访问 `http://localhost:5000` 即可使用。

### 4.3 默认管理员账号

部署完成后，系统自动创建管理员账号：

| 属性 | 值 |
|------|-----|
| 用户名 | `admin` |
| 密码 | `admin123` |

**首次登录后请立即修改密码。**

如需自定义管理员账号，编辑 `.env` 文件中的以下配置：

```env
ADMIN_USERNAME=admin
ADMIN_PASSWORD=admin123
ADMIN_PHONE=13800138000
```

### 4.4 高德地图配置（可选）

平台的地图功能依赖高德地图 API。不配置不影响基本功能使用，但地图相关页面将无法显示。

1. 访问 [高德地图开放平台](https://lbs.amap.com/) 注册账号
2. 创建应用，获取 Web 端 JS API Key
3. 将 Key 填入 `.env` 文件：

```env
AMAP_KEY=你的高德地图Key
AMAP_SECURITY_CODE=你的安全密钥
```

### 4.5 功能操作指南

#### 群众用户

1. **注册登录**：使用手机号注册账号，支持密码登录
2. **浏览悬赏令**：首页查看最新悬赏令，点击查看详情
3. **提交线索**：在悬赏令详情页点击"提交线索"，或直接通过线索举报页面提交。可上传照片、自动定位
4. **上报隐患**：拍照上传隐患照片，选择隐患类型和紧急程度，系统自动获取位置
5. **查询进度**：通过线索编号或匿名查询码追踪处理进度
6. **积分兑换**：在积分商城使用积分兑换商品

#### 志愿者/安保人员

1. **角色升级**：登录后访问"志愿者注册"页面，选择角色类型并填写相关信息
2. **抢单执行**：在任务列表页抢单，按时到达指定地点完成任务并上传证明
3. **巡逻打卡**：巡逻过程中进行位置打卡，保持值班状态
4. **应急响应**：收到应急调度通知后前往现场响应

#### 管理员

1. **登录后台**：使用管理员账号登录，导航栏出现"管理后台"入口
2. **发布悬赏令**：后台创建悬赏令，填写案件信息、设置悬赏金额
3. **处理线索**：审核群众提交的线索，指派民警核查，确认后发放奖励
4. **管理隐患**：查看上报的隐患，标记处理状态
5. **发布任务**：创建巡逻/宣传/应急任务，设置时间、人数和积分奖励
6. **应急调度**：在地图上创建应急事件，调度附近安保力量
7. **用户管理**：管理用户角色、状态，查看积分排行
8. **系统设置**：配置系统参数，管理公告发布

### 4.6 生产环境部署

详细部署步骤请参考 `docs/DEPLOY.md`，包含 Gunicorn + Nginx 的生产环境配置。

## 五、数据库说明

- **本仓库不包含数据库文件**（`*.db` 已加入 `.gitignore`）
- 首次运行 `python run.py` 时，系统自动在项目根目录创建 `app.db` SQLite 数据库文件
- 所有数据表由 SQLAlchemy 自动创建（`db.create_all()`），无需手动执行 SQL
- 管理员账号在应用启动时自动创建（通过 `create_admin()` 函数）
- 如已有数据库文件，系统不会重复创建管理员
